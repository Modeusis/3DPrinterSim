# База знаний и диалог по Creality K1 SE

## Назначение работы
Этот документ нужен для подготовки к объяснению того, как была реализована страница базы знаний в проекте `3DPrinterSim`.

Основная цель работы:
- перенести идею базы знаний из старого проекта;
- адаптировать содержимое под тему `Creality K1 SE`;
- заменить старые триады новыми по теме проекта;
- реализовать диалог с базой знаний без `jQuery`;
- добавить голосовой ввод и автоматическое воспроизведение ответа;
- оформить интерфейс в стиле текущего сайта.

## Какие файлы были изменены

На этой странице размещен интерфейс базы знаний:

- окно истории диалога;
- поле ввода вопроса;

### `Web/js/dialog.js`
В этом файле находится вся логика:
- база знаний;

- TTS-озвучка ответа;
- вывод сообщений в историю;

### `Web/css/style.css`
Здесь размещены стили:

## Что такое база знаний в этом проекте
База знаний хранится как массив триад.

Каждая триада имеет вид:
- подлежащее;
- сказуемое;
- поясняющая часть.

Пример:

```js
['Creality K1 SE', 'имеет', 'кинематику CoreXY']
```

Из таких триад формируются ответы.

Дополнительно рядом с триадами хранится список готовых вопросов. Это сделано для двух задач:
- пользователь может нажать на вопрос и сразу получить ответ;
- программа может быстро найти точный ответ по заранее известной формулировке.

## Что было сделано с содержанием
Изначально в новой версии было слишком мало записей для соответствия методическим требованиям.

Поэтому база была расширена до `108` вопросов и `108` триад.

Содержимое охватывает несколько смысловых блоков:
- основы имитационного моделирования;
- этапы построения модели;
- преимущества и недостатки моделирования;
- история развития 3D-печати;
- устройство FDM-принтера;
- особенности `Creality K1 SE`;
- области применения 3D-печати;
- медиа-ответы с изображением и видео.

## Общий алгоритм работы страницы

### 1. Загрузка страницы
После события `DOMContentLoaded` скрипт:
- получает ссылки на элементы страницы;
- проверяет, что нужные элементы существуют;
- создает массивы вопросов и триад;
- подготавливает словарь точных вопросов;
- выводит список готовых вопросов;
- инициализирует голосовой ввод.

### 2. Формирование списка готовых вопросов
Функция `renderQuestions()`:
- очищает блок вопросов;
- проходит по массиву вопросов;
- для каждого вопроса создает кнопку-карточку;
- показывает краткий фрагмент будущего ответа;
- навешивает обработчик клика.

При клике:
- вопрос подставляется в поле ввода;
- сразу вызывается функция `ask()`.

### 3. Запуск диалога
Когда пользователь:
- нажимает кнопку `Спросить`,
- нажимает `Enter`,
- или выбирает вопрос из списка,

вызывается функция `ask()`.

Она выполняет следующие шаги:
1. получает текст вопроса;
2. если вопрос пустой, завершает работу;
3. останавливает предыдущее голосовое воспроизведение;
4. добавляет сообщение пользователя в историю;
5. вызывает `getAnswer(question)`;
6. добавляет ответ в историю;
7. подключает обработчики для изображения и видео;
8. прокручивает чат вниз;
9. если включена озвучка, запускает TTS.

## Алгоритм поиска ответа
Ключевая функция поиска ответа называется `getAnswer(question)`.

Она работает в несколько этапов.

### Этап 1. Поиск точного совпадения
Сначала вопрос нормализуется:
- переводится в нижний регистр;
- заменяется `ё` на `е`;
- удаляются лишние знаки препинания;
- схлопываются лишние пробелы.

Если после нормализации вопрос совпадает с одним из заранее известных вопросов, ответ берется сразу.

Это самый быстрый и надежный путь.

### Этап 2. Поиск по сказуемому и подлежащему
Если точного совпадения нет, запускается логика семантического поиска:
- вопрос разбивается на слова;
- для каждого слова проверяется, похоже ли оно на сказуемое;
- используются псевдоокончания вроде `ет`, `ит`, `утся`, `ается`;
- если слово подходит, строится регулярное выражение для сказуемого;
- из оставшейся части вопроса строится регулярное выражение для подлежащего.

Дальше программа перебирает все триады и ищет запись, где:
- второе поле подходит под сказуемое;
- первое или третье поле подходит под подлежащее.

Если совпадение найдено, из триады собирается предложение-ответ.

### Этап 3. Резервный поиск по смысловым словам
Если предыдущий этап не дал результата, используется `findBestMatch(question)`.

Эта функция:
- берет значимые слова из вопроса;
- сравнивает их со всеми триадами;
- считает количество совпадений;
- выбирает наиболее подходящую запись.

Если совпадений достаточно, пользователь получает наиболее близкий ответ.

### Этап 4. Сообщение об отсутствии ответа
Если ни один способ не сработал, возвращается фраза:

`Ответ не найден.`

Сейчас она дополнена подсказкой, о чем можно спросить.

## Зачем нужны псевдоокончания
Псевдоокончания нужны для того, чтобы не хранить все формы глаголов вручную.

Например, если пользователь спрашивает:
- `Какая кинематика у K1 SE?`
- `Что поддерживает K1 SE?`
- `Как работает автокалибровка?`

программа может определить сказуемое и найти нужную триаду даже при разных формулировках.

Это делает диалог более гибким.

## Как работает отображение сообщений
История диалога строится не как простой текст, а как набор отдельных блоков сообщений.

Каждое сообщение создается функцией `appendMessage()`:
- создается внешний контейнер сообщения;
- внутри создается пузырек;
- текст вопроса или HTML-ответ вставляется внутрь пузырька;
- сообщение добавляется в блок истории.

Используются два типа сообщений:
- `question__wrapper` для вопроса;
- `answer__wrapper` для ответа.

В стилях это оформлено так:
- вопрос выравнивается влево;
- ответ выравнивается вправо;
- у каждого пузырька есть скругления, отступы и тени.

Именно это делает интерфейс похожим на чат.

## Как работает прокрутка чата
После добавления нового сообщения вызывается `scrollChatToBottom()`.

Функция:
- ждет ближайший кадр браузера через `requestAnimationFrame()`;
- прокручивает блок истории к нижней границе.

Отдельно добавлены обработчики для `img` и `video`, потому что медиа могут догружаться позже и менять высоту блока. После их загрузки скролл тоже повторно уходит вниз.

## Как работает голосовой ввод
Голосовой ввод реализован через:
- `window.SpeechRecognition`
- или `window.webkitSpeechRecognition`

Алгоритм такой:
1. пользователь нажимает кнопку `Голосовой ввод`;
2. браузер начинает распознавание;
3. распознанный текст помещается в поле ввода;
4. вопрос автоматически отправляется в диалог.

Если браузер не поддерживает распознавание речи:
- кнопка отключается;
- пользователю показывается соответствующий статус.

## Как работает озвучка ответа
Озвучка ответа реализована в два слоя.

### Основной вариант
Сначала используется внешний `TTS API`:

- формируется URL с текстом;
- создается объект `Audio`;
- аудио запускается автоматически.

Если ответ длинный, текст режется на части функцией `splitText()`, чтобы не передавать слишком длинную строку одним запросом.

### Резервный вариант
Если внешний TTS недоступен:
- включается `speechSynthesis`;
- создается `SpeechSynthesisUtterance`;
- выбирается русскоязычный голос, если он есть;
- ответ озвучивается средствами браузера.

Такой подход повышает надежность:
- если внешний сервис не ответил, страница не остается без голоса.

## Как работает вывод изображений и видео
Некоторые ответы содержат HTML:
- изображение принтера;
- видео работы принтера.

Для изображений добавлен клик по превью:
- создается затемненный оверлей;
- изображение показывается в увеличенном виде;
- закрытие возможно по кнопке или по клику на фон.

Видео в ответе встраивается прямо в пузырек ответа и может воспроизводиться на месте.

## Почему реализация сделана без jQuery
В старом учебном варианте использовался `jQuery`.

В этой версии от него отказались, потому что требование проекта было:
- не использовать `jQuery`;
- сделать современную реализацию на чистом JavaScript.

Это означает, что вместо `jQuery` применяются:
- `document.querySelector()`;
- `addEventListener()`;
- `appendChild()`;
- `classList`;
- `requestAnimationFrame()`;
- стандартные Web API браузера.

## Возможные вопросы преподавателя и ответы

### Почему используется именно массив триад?
Потому что триада удобно представляет простое знание в форме:
`субъект - связь - пояснение`.

### Почему добавлен точный поиск, если уже есть семантический?
Точный поиск быстрее и стабильнее работает для заранее подготовленных вопросов.

### Зачем нужен резервный поиск?
Чтобы пользователь мог получать ответ даже при неидеальной формулировке.

### Почему добавлен fallback для озвучки?
Потому что внешний TTS может быть временно недоступен, а браузерный синтез повышает надежность.

### Почему интерфейс сделан без jQuery?
Потому что по требованию проекта нужно было реализовать все на нативном JavaScript.
